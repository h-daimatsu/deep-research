# 検証手順（文体 / 引用書式 / 結論 / チェックポイント突合）

レポート出力後に実施する全検証の手順。各検証はステップ14、15、16 として `workflow.md` から参照される。

検証結果はすべて `verification.md` に記録する。

## 検証1: 文体検証（ステップ14）

`report.md` の出力後、`tone-rules.md` に定義された逸脱パターン（読者への問いかけ、著者のメタ自己言及、文学的比喩、判断の完全な保留、修辞的繰り返し、誇張、ベンダー的装飾語の流用）を検出する。

### 手順

本検証は単語のブラックリストによる機械検出ではなく、**文の機能に照らした自己チェック**を主とする。`tone-rules.md` の「回避すべき文の機能」の7カテゴリそれぞれについて、レポート本文を読み返して逸脱がないか確認する。

1. **文の機能別チェック**: レポートを最初から最後まで読み、以下のいずれかに該当する文がないかを確認する。該当があれば中立的な記述に書き換える
   - 機能1: 読者への問いかけ・反語（答えを出さず読者に問いを返す）
   - 機能2: 著者の自己言及・メタ言及（「本レポートが示すのは」「読者に委ねる」等）
   - 機能3: 文学的比喩による説明（抽象概念を具象物に置き換える）
   - 機能4: 判断の完全な保留（「どちらとも言えない」で結論を閉じる）
   - 機能5: 修辞的な繰り返し・対句（効果のための反復）
   - 機能6: 誇張・感情的な強調（数値で裏付けられない強い形容）
   - 機能7: ベンダー的な装飾語の地の文への流用
   - LLM 的な空句: 「重要なのは〜である」「正面から扱う」「不可欠」「核心的」「多角的」「掘り下げる」「〜において」「さらに/また の連打」等（`tone-rules.md`「LLM 的な空句の排除」）。ただしヘッジ語の必要な不確実性は削らない
   - 冗長: 同一主張の言い換え反復、隣接節の役割重複、データ・表の提示直後の要約反復（`tone-rules.md`「冗長の排除」）。深さの削減ではなく重複の削減である点に注意

2. **補助シグナルとしての grep**: 以下は逸脱の兆候を拾う補助的なシグナルであり、ヒットしても必ず逸脱とは限らない（引用内での使用は正当な場合がある）。ヒット箇所は文脈で判断する

   ```bash
   # 機能6（誇張）系の代表的シグナル
   grep -nE '劇的|驚くほど|驚異的|爆発的|画期的|革命的|圧倒的' {フォルダパス}/report.md

   # 機能1（問いかけ・反語）系の代表的シグナル
   grep -nE 'のか。あるいは|のか、それとも|のだろうか' {フォルダパス}/report.md

   # 機能2（メタ自己言及）系の代表的シグナル
   grep -nE '本レポートが示す|読者に委ね|読者が自ら|手元に残す' {フォルダパス}/report.md

   # 機能3（文学的比喩）系の代表的シグナル（文脈判断必要）
   grep -nE '墓標|墓場|街区|重力|対位法' {フォルダパス}/report.md

   # 設計用語の本文への流入
   grep -nE '本レポートの物語|物語軸|ナラティブの力|仮構造|改訂構造' {フォルダパス}/report.md

   # LLM 的な空句系の代表的シグナル（文脈判断必要・ヘッジ語の正当な使用は除外）
   grep -nE '重要なのは|正面から|不可欠|核心的|多角的|包括的|総合的|掘り下げ|深掘り' {フォルダパス}/report.md
   ```

3. 検出・自己チェック結果を `verification.md` の「文体検証」セクションに記録する
4. ヒットした各箇所について、**引用内か地の文か**を判断する
5. 地の文での逸脱は、`tone-rules.md` の7機能の該当項目に沿って中立的記述に書き換える
6. 引用内のヒットは、「〜と表現している」「〜と述べている」の形で引用として切り出されているか確認する
7. 修正後に再度1〜2を実施し、地の文での逸脱がなくなるまで繰り返す
8. 地の文での残逸脱がゼロになった状態で、次の検証2に進む

## 検証2: 引用書式検証（ステップ15）

文体検証の後、最終コンソール出力に進む前に、脚注記法の書式違反と対応漏れを機械スキャンする。

### 手順

1. **禁止形（旧方式）の残存検出**:

   ```bash
   grep -nE '\[[0-9]+\]\(http|\[\[[0-9]+\]\]\(|&#91;|&#93;' {フォルダパス}/report.md
   ```

   ヒットがあれば脚注記法に置換する:
   - `[N](URL)` → 本文を `[^N]` にし、URL は末尾の `[^N]: URL` へ移動
   - `[[N]](URL)` → 同上
   - `[&#91;N&#93;](URL)` / `&#91;N&#93;` → 同上

2. **本文中の素 URL 検出**（脚注を使わず直接 URL を書いている箇所）:

   ```bash
   grep -nE '(^|[^\("<])(https?://[^\s\)]+)' {フォルダパス}/report.md
   ```

   末尾の脚注定義行（`[^N]: ...`）にある URL は正当。本文中に直接現れた URL は脚注化する

3. **孤立参照の検出**（本文に `[^N]` があるのに末尾で定義されていない）:

   ```bash
   # 本文で使われている番号の集合
   refs=$(grep -oE '\[\^[0-9]+\]' {フォルダパス}/report.md | grep -v ':' | sort -u)
   # 末尾で定義されている番号の集合
   defs=$(grep -oE '^\[\^[0-9]+\]:' {フォルダパス}/report.md | sort -u)
   # 両者を比較して不一致を特定
   diff <(echo "$refs" | sed 's/$/:/') <(echo "$defs")
   ```

4. **未使用定義の検出**（末尾で `[^N]:` が定義されているのに本文で使われていない）: 上記 diff の出力で判定

5. **番号の欠番・重複検出**: 定義側の番号が 1 から連続しているか、同じ番号の定義が重複していないかを確認

6. 検出結果を `verification.md` の「引用書式検証」セクションに記録する

7. すべての違反を修正し、再スキャンでヒットゼロ（禁止形）・孤立参照ゼロ・未使用定義ゼロ・欠番/重複ゼロを確認する

8. その状態で `report.md` を再度コンソールに出力する（修正後の全文出力）

## 検証3: チェックポイント突合 + 結論検証（ステップ16）

レポート完成後、`checkpoint.md` の各項目が `report.md` に反映されているか、および結論セクションの5要素が充足されているかを検証する。

### 手順

1. `checkpoint.md` を読み直す
2. 各ステップの「主要な発見」がレポートに含まれているか確認する
3. 各ステップの「因果関係の分析」「波及効果」がレポートに反映されているか確認する
4. 「注目すべきエピソード・発言」がレポートで活用されているか確認する
5. 漏れがあればレポートに追記する
6. **結論セクションの妥当性を検証する**:
   - エグゼクティブサマリーで提示した主題に対する明確な回答があるか
   - 主要な発見（2〜3点）の再提示があるか
   - 誰にとって何を意味するかの含意が述べられているか
   - 読者が取るべきアクションまたは注視ポイントが示されているか
   - 短期〜中期の展望が示されているか
   - 独自の洞察との内容重複が過剰でないか（結論は統合的・処方的であること）
   - テーマの性質に応じた重点要素（`conclusion-guide.md` の「結論の重点はテーマに応じて動的に決める」参照）が厚く書かれているか
   - いずれかが欠けていればレポートに追記する
7. **構造設計との整合性検証**: `structure.md` の改訂後の章立てと `report.md` の実際の章立てが一致しているか、改訂理由が実現されているかを確認する
8. **展開深度の確認（軽量）**: 本検証パスのついでに、柱の発見が含意（だから何）まで展開されているか、事実の列挙や表の投げ放しで打ち切られていないかを目視確認する（`output-calibration.md`）。**明らかな打ち切りを見つけた箇所のみ補筆する。スコアリングや全文再生成は行わない**

## 検証4: provenance・確信度検証（ステップ17）

チェックポイント突合の後、柱の主張の確からしさが正しく表現されているかを検証する。これは「源泉が揃っていても怪しい不確実性（単一源泉・一次未確認・横並び誤報）」を炙り出すための検証である。

### 手順

1. `checkpoint.md` の「柱の主張のprovenance」を読み直し、結論を支える柱の主張を一覧化する
2. 各柱について、`report.md` の対応する脚注定義に確信度タグ（`〔一次照合〕` / `〔二次・独立複数〕` / `〔要確認: 単一源泉/一次未確認〕`）が付いているか確認する。付いていなければ付与する
3. **independent count の検証**: 複数の脚注が実は同一の一次源に畳まれる（偽のコンセンサス）箇所がないか確認する。畳んで独立源泉が1つになる柱は `〔要確認〕` に修正する
4. **本文ヘッジの検証**: `〔要確認: 単一源泉/一次未確認〕` の柱が、本文で断定口調になっていないか確認する。断定があれば限定詞（ヘッジ語）に書き換える（`tone-rules.md`）
5. **不確実性章との整合**: `〔要確認〕` の柱が「情報の矛盾・不確実な点」に列挙されているか、事実抽出系で一次未確認の柱が「追加調査が必要な領域」に挙がっているか確認する
6. 検出・修正結果を `verification.md` の「provenance・確信度検証」セクションに記録する

## verification.md のテンプレート

```markdown
# 最終突合結果

- チェックポイント項目数: {N}
- レポート反映済み: {N}
- 漏れ発見・追記: {N}
- 文体逸脱（地の文）: {N}件 → {N}件修正済
- 引用書式違反: {N}件 → {N}件修正済
- 結論セクションの5要素: {N}/5 充足
- 構造設計との整合性: {一致 / 差異あり}
- 展開深度の打ち切り: {N}件補筆
- 柱の主張の確信度タグ付与: {N}/{N} 充足 — 〔要確認〕の本文ヘッジ化: {N}件 → {N}件修正済

## 詳細
| ステップ | 主要発見 | 因果分析 | 波及効果 | エピソード | 判定 |
|---|---|---|---|---|---|
| 1 | ✅ | ✅ | ✅ | ✅ | OK |
| 2 | ✅ | ⚠️追記 | ✅ | — | 修正済 |

## 結論セクション検証
| 要素 | 有無 | 備考 |
|---|---|---|
| 主題への回答 | ✅ | エグゼクティブサマリーで提示した主題に対応 |

## 構造設計との整合性検証
| structure.md の章 | report.md の対応 | 判定 |
|---|---|---|
| {章タイトル1} | ✅ 一致 | OK |
| {章タイトル2} | ⚠️ 見出し微修正 | 許容 |
| {改訂で追加した章} | ✅ 反映済み | OK |

## 文体検証
| 箇所 | 原文 | 該当機能 | 判定（地の文/引用内） | 修正後 |
|---|---|---|---|---|
| 2章 | 劇的に改善 | 機能6: 誇張 | 地の文 | 40%改善 |
| 3章 | 〜と表現している「驚異的な結果」 | 機能6: 誇張 | 引用内 | 維持 |

## 引用書式検証
| 種別 | 箇所 | 原文 | 修正後 |
|---|---|---|---|
| 禁止形（旧インラインリンク） | 1章 | `事実 [3](https://example.com)` | `事実 [^3]` ＋ 末尾に `[^3]: ...` |
| 素URL | 2章 | `本文中に https://example.com/xxx が露出` | 脚注化（`[^N]` を採番して末尾定義へ） |

## provenance・確信度検証
| 柱の主張 | タイプ | independent count | 一次照合 | タグ | 本文ヘッジ | 判定 |
|---|---|---|---|---|---|---|
| 議決権85.1% | 事実抽出系 | 1 | 未 | 〔要確認〕 | ⚠️→修正 | 修正済 |
| 需給が価格を支える | 解釈系 | 3 | — | 〔二次・独立複数〕 | — | OK |
```

最終検証時の残ヒット件数（禁止形/素URL/孤立参照/未使用定義/欠番/重複）はすべて 0 であること。
