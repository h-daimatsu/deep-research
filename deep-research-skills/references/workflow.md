# ワークフロー（フェーズ1〜5）

Deep Research スキルのメインワークフロー。固定アーキタイプを使わず、テーマごとにレポート構造を動的に設計する。

関連リファレンス:
- 構造設計の原則: `structure-design-guide.md`
- 共通骨格: `common-skeleton.md`
- 引用書式: `citation-format.md`
- 文体ルール: `tone-rules.md`
- 結論の書き方: `conclusion-guide.md`
- 検証手順: `verification.md`

## フェーズ1: 調査計画と仮構造の設計

### ステップ1: テーマの分析

ユーザーのリサーチテーマを受け取ったら、以下を行う。

1. 出力先フォルダを作成する（`{YYYYMMDD-HHmm}-deep-research-{テーマの短縮名}/`）
2. テーマの核心を特定する
3. テーマを3〜7個の調査ステップに分解する
4. 各ステップの以下を整理する:
   - 調査優先度（高/中/低）
   - 依存関係（前のステップの結果がどう影響するか）
   - 調査の目的（何を明らかにするか）

**分解の指針:**
- 「現状の事実」「原因・背景」「影響・波及効果」「リスク・不確実性」「将来の展望」の5軸を意識する
- テーマの性質に応じて軸を取捨選択する
- 固定テンプレートに当てはめず、テーマ固有の論点を抽出する

### ステップ2: 仮構造の設計

調査計画に基づいて、レポートの仮構造を設計する。`structure-design-guide.md` の原則に従う。

設計する内容:
1. **中心主題**: レポート全体を貫く問い（「なぜこれが重要なのか」「何が構造的に変わっているのか」等）
2. **章立て**: 各章の見出し、その章が答える問い、前後の章との接続
3. **分析要素**: 各章で使う分析手法（因果分析、比較表、定量データ、ステークホルダー分析等）
4. **定量/定性の比率**: テーマの性質に応じた比率の見積もり
5. **共通骨格の配置**: `common-skeleton.md` の必須要素をどこに置くか

設計した仮構造を `structure.md` に記録する。

```markdown
# レポート構造設計

## 中心主題
{レポート全体を貫く問い}

## 仮構造（調査前）

### 設計根拠
{なぜこの構造にしたか — テーマの性質、想定される論点、読者が必要とする情報の流れ}

### 章立て
1. エグゼクティブサマリー（共通骨格）
2. {章タイトル} — {この章が答える問い}
   - 分析要素: {因果分析 / 比較表 / 定量データ等}
   - 前章との接続: {前章の結論をどう受けるか}
3. {章タイトル} — {この章が答える問い}
   ...
N-3. 情報の矛盾・不確実な点（共通骨格）
N-2. 追加調査が必要な領域（共通骨格）
N-1. 結論（共通骨格）
N. 出典（共通骨格）

### 定量/定性の比率
{例: 定量 40% / 定性 60% — 理由}
```

### ステップ3: 調査計画と仮構造の提示・承認

todo_list ツールで調査計画を作成し、ユーザーに仮構造とセットで提示する。

```
【調査計画】
テーマ: {テーマ}
出力先: {フォルダパス}

中心主題: {レポート全体を貫く問い}

レポート構造（仮）:
  1. エグゼクティブサマリー
  2. {章タイトル} — {この章が答える問い}
  3. {章タイトル} — {この章が答える問い}
  ...
  N. 結論 / 出典

  設計根拠: {なぜこの構造にしたか}
  定量/定性: {比率と理由}

調査ステップ:
  1. [高] {何を調べ、何を明らかにするか}
  2. [高] {(1)の結果を踏まえて、何を深掘りするか}
  3. [中] {何と何を比較・評価し、どんな判断材料を得るか}
  ...

→ 全調査完了後、結果に基づいて構造を改訂し、レポートを作成

この計画と構造で進めてよいですか？
構造の変更希望があれば指定してください。
```

ユーザーの承認を得てからフェーズ2に進む。

## フェーズ2: 反復的調査

各調査ステップに対して以下のループを実行する。

### ステップ4: 一次情報の探索

テーマに関する一次情報（公式発表、原典、当事者による発信）を探す。

- 公式サイト、プレスリリース、公的機関の発表
- 学術論文、ホワイトペーパー、技術仕様書
- 当事者のブログ、インタビュー、カンファレンス発表
- テーマに応じた一次ソース（IR 資料、官報、技術仕様書等）

一次情報が見つかった場合は Fetch ツールで詳細を取得する。**`tavily_extract` が利用可能なら主軸として使い、標準の Web fetch ツールは補完用**（詳細は下記の Fetch ツールの運用ルール参照）。

#### 主張のタイプ別の確からしさ戦略（事実抽出系 vs 解釈系）

主張のタイプによって、何が確からしさを担保するかが異なる。混同すると「エンジンを増やせば確かになる」という誤認に陥る。

| 主張のタイプ | 例 | 確からしさを担保するもの | 一次照合 |
|---|---|---|---|
| **事実抽出系** | 正確な財務数値、議決権比率、契約金額・文言、株式数 | **原本そのもの**。二次のコンセンサスは単なるエコーで、エンジン多様化は無力 | **必須**。原本未照合なら独立検証ゼロとみなす |
| **集合知・コンセンサス系** | アナリストコンセンサス（売上/EPS予想）、目標株価の平均・レンジ、市場規模・シェア推計、各社ガイダンスの集計 | **集計それ自体が情報**。多数の独立した予測の分布（中央値・レンジ・件数）が価値を持ち、単一の一次原本は存在しない | **概念上不要**。集計元が複数の独立機関であることを確認すれば足りる |
| **解釈・市場観系** | 「需給が支える」「ガバナンス・ディスカウント」等の評価 | **複数アウトレットの多様性**が価値を持つ | 不要。独立した複数の論者の併記で担保 |

主張のタイプを取り違えると確信度の評価を誤る。**事実抽出系**は原本に到達できなければ、何本の二次記事が一致していても確信度は上がらない（矛盾の有無に関係なく〔要確認: 一次未確認〕とする）。一方、**集合知・コンセンサス系と解釈・市場観系は、複数の独立したソース・論者の一致がそのまま確信度を上げる**。これらに事実抽出系の「一次未確認＝低確信」基準を機械的に適用してはならない。アナリストコンセンサスや市場規模推計を「一次原本がないから〔要確認〕」と扱うのは過剰な留保であり、レポートの情報価値を不当に損なう。集合知・解釈系で独立複数ソースが一致するものは `〔二次・独立複数〕` として通常の確信度で本文に記述してよい。留保が必要なのは、源泉を畳むと独立ソースが実は1つ（偽のコンセンサス）の場合に限る。

#### 一次照合のフォールバック（原本に到達できないとき）

事実抽出系の柱について、一次原本（SEC EDGAR、官報、IR 開示、論文等）への到達を試みる。

**到達手段の優先順位（実測知見）:**

1. **`tavily_extract`（advanced）を第一手にする**: SEC EDGAR の文書ホスト（`www.sec.gov/Archives/...`）は標準 Web fetch では**確定的に `403 Forbidden`** を返す。標準 fetch の 403 を一次情報の不在と誤認しない。`tavily_extract` は JS レンダリングとボイラープレート除去で原本本文に到達できることが多い
2. **代替エンドポイントを使う**: HTML 版が取れない場合、EDGAR full-text search（`efts.sec.gov`）や `data.sec.gov` の JSON、文書インデックスページ経由で到達できる場合がある
3. **別の検索エンジンで原本の別ホスト・ミラーを探す**

**目的の柱が取れないときの再クエリ（省略不可）:**

`tavily_extract` はクエリ関連チャンクをリランクして返すため、**巨大原本（S-1 等）では1クエリで全セクションは返らない**。狙った事実抽出系の柱（例: 報酬株式数、訴訟権放棄条項）が返らなければ、その柱固有の語で再クエリしてから判定する。柱ごとに最低1回は固有語での再取得を試み、それでも取れない場合に限り `〔要確認: 一次未確認〕` とする。1クエリの空振りを「原本に記載なし」と早合点しない。

それでも到達できなければ、当該の柱を「追加調査が必要な領域」と「情報の矛盾・不確実な点」の両方に明記し、本文ではヘッジ語に統一する（断定しない）。

#### provenance collapse（源泉まで畳む）

柱の主張について、記事単位ではなく**一次源単位**で独立ソース数を数える。

1. 各柱の主張に「最終的にどの一次情報に由来するか（primary origin）」を1行記録する
2. 二次記事を origin に畳んで重複排除する（「Reuters 報道を引用した15本」は independent count = 1）
3. independent count = 1 かつ事実抽出系で一次未確認なら、高不確実度フラグを立てる
4. **echo を検出したら原本を取りに戻る（省略不可）**: independent count = 1 で、かつその一次源が到達可能な原本（EDGAR、官報、IR 開示等）の場合、二次コンセンサスで確定させる前に、その数値・文言固有の語で一次抽出を最低1回試みる（上記「目的の柱が取れないときの再クエリ」を適用）。原本照合に成功すれば `〔一次照合〕` に格上げし、失敗して初めて `〔二次〕`／`〔要確認: 一次未確認〕` に留める。「複数の二次が一致している」ことを原本照合を省く理由にしない

このレイヤーは retrieval の保険（複数エンジン併用）の上に乗せる二層構成とし、見かけのコンセンサスに騙される率を下げる。

#### 検索ツール: 複数エンジンの併用

Web 検索に使えるツールが複数ある場合（例: 標準の Web 検索ツール / `tavily` / `brave` が利用可能な環境）、**同一または類似のクエリを複数のツールに投げて結果を突合する**。検索エンジンごとにインデックスやランキングの傾向が異なり、単一エンジンでは到達できないソースが存在するため、複数併用は**取得（retrieval）の網羅性**を高める手段である。ただしこれは取得の多様化であって**源泉（source）の多様化ではない**——3エンジンが返した記事群が全て同一の一次情報に行き着くなら独立源泉は1つであり、本数の多さは確からしさではない（後述の provenance collapse 参照）。

**運用ルール:**

1. **利用可能な検索ツールを確認する**: 環境によって使えるツールは異なる。利用可能なツールが1つしかなければ、そのツールのみで調査を進める
2. **複数ツールがある場合、各調査ステップの初回検索は複数エンジンで並行実行する**: 同一クエリでも、エンジンごとに異なるソースが返る。結果をマージしてソースの多様性を確保する
3. **良質な一次ソースが見つかったら、追加検索より全文取得・精読を優先する**: ソース数を増やすことよりも、1つの良質な一次ソース（当事者インタビュー、長編特集記事等）から複数の情報を引き出す方がレポート品質に寄与する。Fetch ツールで記事を精読し、発言・数値・エピソードを徹底的に抽出する
4. **ユーザーが検索ツールを指定した場合は、その指定に従う**

**検索の「幅」と「深さ」のバランス:**

検索エンジンの併用はソースの多様性を高めるが、コンテキストの消費も増える。有限のコンテキストの中で、検索の幅を広げれば深掘りの余裕が減る。このトレードオフを意識し、以下の優先順位で行動する。

1. **初回検索で幅を確保する**（複数エンジン並行実行）
2. **良質なソースを発見したら深さに切り替える**（全文取得・精読・情報抽出）
3. **情報が不足している場合のみ、追加の幅を取る**（別エンジンでの補完検索）

「10本の記事を浅く読む」より「3本の良質な記事を徹底的に読む」方が、分析の深さと具体性においてレポート品質を高める。

#### Fetch ツール: 主従使い分け

Fetch ツールは検索ツールと違って**並行併用しない**。同じページを複数ツールで取っても基本的に同じ内容が返るだけで、コンテキストと API コールの二重消費になる。機能セットの差を踏まえて主従を決める。

**主軸: `tavily_extract`（利用可能な場合）**

- JS レンダリング対応（`advanced` モード）
- ボイラープレート（広告・ナビ・フッター）除去済み markdown
- 最大 20 URL の一括取得
- クエリを渡すと関連チャンクをリランク（query-focused chunking）

**補完: 標準の Web fetch ツール**

以下のいずれかに該当するときに使う。

| 場面 | 理由 |
|---|---|
| 長文ページから特定フレーズ周辺だけ欲しい | `selective` モードでピンポイント抽出 |
| PDF・プレーンテキスト等のシンプルなソース | 整形不要なら標準の Web fetch ツールで十分 |
| 同一ドキュメントの続きを読む（ページネーション） | `start_index` で続きを取得できる |
| `tavily_extract` が空・エラーで返った | フォールバック。`rendered` モードで再試行 |

**フォールバック順:**

1. `tavily_extract`（基本）→ 内容が薄い・空
2. `tavily_extract`（advanced モードに切り替え）→ それでも取れない
3. 標準の Web fetch ツール（truncated 相当モード）→ 静的取得を試す
4. 標準の Web fetch ツール（rendered 相当モード）→ JS 必須サイトの最終手段

`tavily_extract` が利用できない環境では標準の Web fetch ツールのみで運用する。ユーザーが Fetch ツールを指定した場合はその指定に従う。

### ステップ5: 二次情報による補完

一次情報だけでは不十分な場合、二次情報で補完する。

1. 複数の独立したソースから情報を収集する
2. ソース間の一致点と相違点を記録する
3. 情報の鮮度（公開日・更新日）を確認する
4. **利用可能な検索ツールが複数ある場合、ステップ4で使わなかったツールでも補完検索を行う**

### ステップ6: 深掘り分析

収集した情報に対して、以下の3つの問いを立てる。

1. **「なぜ？」（因果関係）**: この事実はなぜ起きているのか
2. **「だから何？」（波及効果）**: この事実は他の領域にどう影響するか
3. **「本当に？」（検証）**: この主張は他のソースでも裏付けられるか

これらの問いから新たな検索キーワードを生成し、追加検索を実行する。

### ステップ7: 情報飽和の判定

以下のいずれかを満たすまで、ステップ4〜6のループを繰り返す。

- **飽和**: 追加検索で新しい事実や視点が得られなくなった
- **十分性**: 調査ステップの目的に対して必要な情報が揃った
- **上限**: 同一ステップで5回以上検索しても飽和しない場合は、不足を明記して次に進む

### ステップ8: チェックポイントの記録

**各調査ステップの完了時に、必ず `checkpoint.md` に追記する。省略不可。**

```markdown
## ステップN: {ステップのテーマ}（完了: {日時}）

### 主要な発見
- {発見1}（ソース: {URL}）
- {発見2}（ソース: {URL}）

### 定量データ（該当する場合）
{数値、統計、比較データを表形式で}

### 因果関係の分析
{なぜそうなっているか}

### 波及効果
{他の領域への影響}

### 矛盾・不確実な点
{ソース間の矛盾、未確認事項}

### 柱の主張のprovenance（確からしさの検証）
{結論を支える主張のみ。枝葉は不要}
- 主張: {柱の主張} / タイプ: {事実抽出系 / 解釈系} / primary origin: {最終的な一次源} / independent count: {源泉を畳んだ後の独立ソース数} / 一次照合: {済 / 未} → 確信度: {一次照合 / 二次・独立複数 / 要確認: 単一源泉・一次未確認}

### 次のステップへの影響
{この発見が後続ステップの方向性をどう変えるか}

### 仮構造への示唆
{この発見が仮構造の妥当性にどう影響するか — 章の追加・削除・統合・分割の候補}

### 注目すべきエピソード・発言
{レポートで使えそうな具体的な発言や出来事}
```

チェックポイント記録後、todo_list の該当タスクを完了にする。

### ステップ9: 進捗報告

チェックポイント記録後、簡潔な進捗をユーザーに報告する。

```
✅ ステップN完了: {主要な発見の1行要約}
  発見した新たな論点: {あれば記載}
  → 仮構造への示唆: {構造改訂で検討すべき点があれば記載}
→ ステップN+1の調査を開始します
```

調査中に以下があればユーザーに提案する:
- 調査計画の修正（スコープの絞り込み等）

## フェーズ3: 構造の改訂

全調査ステップ完了後、仮構造を改訂する。**このフェーズは省略不可。**

> **多段推論の再点火（構造改訂・統合分析の前に適用）**: This task involves multi-step reasoning. Think carefully through the problem before responding. 本フェーズは分析の深さ・独自の洞察が最も生まれる工程である。構造の改訂と統合分析に着手する前に、各発見の因果チェーン（事実A→事実B→影響C）と、複数の発見を組み合わせて初めて見える横断的な解釈を段階的に考え抜く。個別事実の列挙で止めず、それらの相互作用・メタ洞察まで推論を進めてから改訂・統合に入る。

### ステップ10: チェックポイントの読み直し

**必ず `checkpoint.md` を全文読み直す。** コンテキストウィンドウの制約で初期の調査結果が失われている可能性があるため、省略不可。

読み直した上で、以下を確認する:
- 各ステップの主要な発見がすべて把握できているか
- ステップ間の接続（因果関係、相互作用）が明確か
- 各ステップの「仮構造への示唆」を集約する

### ステップ11: 構造の改訂

仮構造を、調査結果に基づいて改訂する。`structure-design-guide.md` の原則に再度照らす。

改訂で検討する項目:
1. **章の追加**: 調査で予想外の重要論点が見つかった場合
2. **章の削除**: 情報が不十分で薄い章になる場合、他の章に統合する
3. **章の分割**: 1章の情報量が多すぎる場合
4. **章の順序変更**: 調査結果から、より自然な因果の流れが見えた場合
5. **分析要素の変更**: 当初想定と異なるデータや視点が得られた場合
6. **定量/定性比率の調整**: 実際に得られたデータの量に応じて
7. **中心主題の微調整**: 調査結果が当初の問いを修正する場合

改訂結果を `structure.md` に追記する。

```markdown
## 改訂構造（調査後）

### 改訂理由
{仮構造からの変更点とその根拠を箇条書きで}

### 改訂後の章立て
1. エグゼクティブサマリー（共通骨格）
2. {章タイトル} — {この章が答える問い}
   ...
```

### ステップ12: 統合分析

すべての調査結果を統合し、以下を行う。

1. **中心主題の確定**: 改訂後の中心主題を最終確定する。本文では「本レポートの主題」等の中立表現で参照する
2. **因果チェーンの構築**: 事実A→事実B→影響Cの連鎖を明示する
3. **セクション間の接続を設計する**: 各セクションの結論が次のセクションの前提になるよう論理の流れを設計する
4. **対立する見方の整理**: 楽観的シナリオと悲観的シナリオの両方を整理する
5. **独自の洞察の導出と配置設計**: 事実を組み合わせて得られる新しい解釈や示唆を整理する。**洞察は独立セクションに分離せず、関連する本体セクションの分析に織り込む**。洞察が本文の論理展開の一部として機能し、読者が自然に到達できる構成にする。独立した「洞察セクション」を設けるのは、本体セクションのどこにも収まらない横断的な解釈がある場合のみとする
6. **エピソードの選定**: 関係者の発言、具体的な出来事を調査結果から選び出す。分析的な引用（アナリストの評価、専門家のコメント）は、事実の裏付けだけでなく、分析の視点を補強する素材として積極的に活用する
7. **結論の骨子を設計する**: `conclusion-guide.md` の5要素に沿って整理する

## フェーズ4: レポート作成

### ステップ13: レポート出力

改訂後の構造に従って `report.md` を作成する。

**共通骨格（必須）:**

`common-skeleton.md` に定義された要素を必ず含める:
1. `## エグゼクティブサマリー`（冒頭）
2. 動的に設計した本体セクション群
3. `## 情報の矛盾・不確実な点`
4. `## 追加調査が必要な領域`
5. `## 結論`（`conclusion-guide.md` の5要素）
6. `## 出典`（`citation-format.md` の脚注定義形式）

**レポートの書き方:**

- エグゼクティブサマリーで中心主題を提示し、読者に全体像を与える
- 各セクションは前のセクションの結論を受けて展開する
- データや数字は「それが何を意味するか」の解釈を散文とセットで提示する
- 関係者の発言や具体的な事実を織り交ぜ、分析に事実性を持たせる
- 引用は `citation-format.md` に従う
- 文体は `tone-rules.md` に従う
- 結論セクションは `conclusion-guide.md` の5要素を必ず含める

**出力後の操作:**

ファイル保存後、`report.md` の内容を省略せずそのままコンソールにも出力する。出力後に `📄 レポートを出力しました: {フォルダパス}/report.md` を表示する。

## フェーズ5: 最終突合

全検証の詳細な手順は `verification.md` を参照する。

### ステップ14: 文体検証（省略不可）

`tone-rules.md` の逸脱パターンを検出・修正する。`verification.md` の「検証1: 文体検証」の手順に従う。

### ステップ15: 引用書式検証（省略不可）

脚注記法の書式違反と対応漏れを機械スキャンする。`verification.md` の「検証2: 引用書式検証」の手順に従う。修正後に `report.md` を再度コンソールに出力する。

### ステップ16: チェックポイントとレポートの突合

`verification.md` の「検証3: チェックポイント突合 + 結論検証」の手順に従う。

追加の検証項目（本スキル固有）:
- **構造の改訂が反映されているか**: `structure.md` の改訂後の章立てと `report.md` の実際の章立てが一致しているか
- **改訂理由が実現されているか**: 改訂で追加・削除・統合した章が、意図通りにレポートに反映されているか

### ステップ17: provenance・確信度検証（省略不可）

`verification.md` の「検証4: provenance・確信度検証」の手順に従う。柱の主張に確信度タグが付いているか、〔要確認〕の柱が本文でヘッジ語になっているか、独立ソース数が脚注から追えるかを確認する。

検証結果はすべて `verification.md` に記録する。突合完了後、最終的なレポートのパスをユーザーに通知する。
