# レポート構造の設計原則

Deep Research スキルでは、固定テンプレートを使わず、テーマごとにレポート構造を動的に設計する。本ドキュメントは、その設計の原則を定める。

ワークフローのステップ2（仮構造の設計）とステップ11（構造の改訂）で参照する。

## 設計の前提

レポート構造とは、以下の要素の組み合わせである:

- **章立て**: 各章の見出しと順序
- **各章が答える問い**: その章が読者のどんな疑問に答えるか
- **章間の接続**: 前の章の結論が次の章の前提になる論理の流れ
- **分析要素**: 各章で使う分析手法（因果分析、比較表、定量データ等）
- **定量/定性の比率**: テーマの性質に応じたバランス

構造設計は「テンプレートを選ぶ」作業ではなく、「テーマに最適な論理の流れを設計する」作業である。

## 原則1: 各章は独立した問いに答える

各章には、その章が答える明確な問いがなければならない。

- 良い例: 「サプライチェーンの現状はどうなっているか」「地政学的リスクはどの類型に分けられるか」
- 悪い例: 「関連情報」「その他の論点」「補足」

問いが立てられない章は、他の章に統合するか、削除を検討する。

## 原則2: 章の順序は論理で決める

章の並べ方には以下のパターンがある。テーマに最も自然なパターンを選ぶ。

| パターン | 適するテーマ | 例 |
|---|---|---|
| **因果連鎖** | 原因→結果→影響を追うテーマ | 原因の分析 → 直接的影響 → 二次的波及 → 対応策 |
| **時系列** | 経緯や変遷を追うテーマ | 背景 → 発端 → 展開 → 現状 → 展望 |
| **構造分解** | 複合的な対象を分解するテーマ | 全体像 → 要素A → 要素B → 要素C → 統合 |
| **比較軸** | 複数の対象を比較するテーマ | 評価基準の定義 → 対象A → 対象B → 比較分析 → 選定 |
| **ステークホルダー別** | 影響が多方面に及ぶテーマ | 全体像 → 消費者への影響 → 企業への影響 → 規制当局の対応 |
| **スケール別** | ミクロからマクロ（またはその逆）に展開するテーマ | 個別事例 → 業界動向 → マクロ経済への影響 |

**複数パターンの組み合わせも可**。例えば、前半は時系列で経緯を追い、後半はステークホルダー別に影響を分析する構成。

**避けるべき順序:**
- 並列的な羅列（章同士に論理的な接続がない）
- 重要度順の単純な並べ替え（論理の流れが生まれない）

## 原則3: 章間の接続を明示する

各章は前の章の結論を受けて展開する。設計時に、章間の接続を明示的に記述する。

- 「第2章で明らかになった市場構造の変化が、第3章で分析する企業戦略の背景となる」
- 「第3章の技術的制約が、第4章のリスク分析の前提条件を形成する」

接続が設計できない場合、章の順序が不適切であるか、章が独立しすぎている可能性がある。

## 原則4: 分析要素はテーマに応じて選択する

以下は使用可能な分析要素の一覧である。すべてを使う必要はなく、テーマに適したものを選ぶ。

### 定性的な分析要素

| 要素 | 用途 | 適するテーマ |
|---|---|---|
| **因果分析** | 事実Aが事実Bを引き起こす構造の解明 | 戦略、政策、事件の背景分析 |
| **ステークホルダー分析** | 関係者ごとの立場・影響・利害の整理 | 政策、規制、業界再編 |
| **シナリオ分析** | 楽観/悲観/基本シナリオの提示 | 将来予測、リスク評価 |
| **フレームワーク適用** | 既存の分析枠組みの適用 | 競争環境、イノベーション、市場構造 |
| **対立する見方の併記** | 評価が分かれる論点の両論提示 | 論争的なテーマ全般 |
| **事例分析** | 具体的な事例の詳細な検討 | 成功/失敗パターン、ベストプラクティス |

### 定量的な分析要素

| 要素 | 用途 | 適するテーマ |
|---|---|---|
| **データ表** | 数値の時系列推移、比較 | 財務、市場規模、性能指標 |
| **比較マトリクス** | 複数対象の多軸比較 | 技術選定、競合比較 |
| **指標の推移** | KPI や市場指標の変化 | 投資、市場動向 |

### フレームワークの例

テーマに応じて以下のフレームワークの適用を検討する。無理に当てはめる必要はない。

- ポーターの5フォース（競争環境）
- SWOT 分析（戦略的ポジション）
- PESTLE 分析（マクロ環境）
- ジェボンズのパラドックス（効率化と需要の関係）
- イノベーションのジレンマ（既存企業の対応）
- ゲーム理論的分析（複数プレイヤーの戦略的相互作用）

## 原則5: 定量/定性の比率はテーマで決まる

テーマの性質に応じて、定量データと定性分析の比率を設計時に見積もる。

| テーマの性質 | 定量の目安 | 定性の目安 | 例 |
|---|---|---|---|
| 数値判断が中心 | 60〜70% | 30〜40% | 投資判断、市場規模分析 |
| 構造分析が中心 | 20〜40% | 60〜80% | 戦略分析、政策影響分析 |
| バランス型 | 40〜50% | 50〜60% | 技術評価、競合比較 |

この比率は仮構造の段階で見積もり、調査後に実際のデータ入手状況に応じて改訂する。

**定量データの密度を意識的に確保する**: 定性分析が中心のテーマであっても、各章に最低1つは具体的な数値・データ・比較表を含めることを目標とする。定量データは分析の説得力を裏付ける。数値なしの章が続くと、主張が根拠薄弱に見える。調査段階で以下を意識的に収集する:

- 規模を示す数値（市場規模、シェア、ユーザー数、投資額）
- 変化を示す数値（成長率、前年比、推移）
- 比較を示す数値（競合との差、ベンチマーク）
- 計画を示す数値（目標値、期限、マイルストーン）

## 原則6: 章の粒度を揃える

各章の情報量は概ね均等にする。

- **1章が肥大化したら分割を検討する**: 1章の中に複数の独立した問いが含まれている場合は分割する
- **1章が薄すぎたら統合を検討する**: 十分な情報が得られなかった章は、関連する章に統合する
- **目安**: 本体セクションは3〜7章（H2）。2章以下は浅すぎる。章数（H2）を増やしすぎず、複雑テーマの厚みは次の「2階層展開」で確保する

### 複雑テーマは2階層で展開する（章→小節）

1つの章が複数の列挙可能な対象・要因・セグメントを扱う場合、それらを密な段落や表の各行に圧縮せず、**H3小節に分解して各々を展開する**。これが複雑テーマの厚みの主因である。

- 例: 候補が4分類 → 各分類をH3小節で個別評価／構造変化が6要因 → 各要因をH3小節で論じる／対象者が5タイプ → 各タイプをH3小節で推奨提示
- 「3〜7章」の目安はH2（章）の数である。複雑テーマの厚みはH2を増やすのではなく、**主要章をH3小節へ分解する**ことで確保する。複雑テーマ（調査ステップ5〜7規模）では、主要章がH3で分解されているのが標準的な姿である
- **仮構造（`structure.md`）の設計段階でH3小節まで列挙し、ユーザー承認を得る**。書き始めてから小節を足すのではなく、構造設計時に粒度を決める
- 単純テーマ（調査ステップ3規模）では無理にH3へ分解しない。各小節は固有の発見を持つ場合にのみ立て、中身のない小節を量産しない
- 詳細は `output-calibration.md`「構造の粒度とスコープ」

## 原則7: 共通骨格は必ず組み込む

`common-skeleton.md` に定義された必須要素は、どんな構造でも必ず含める。動的に設計するのは共通骨格の「間」に入る本体セクションである。

```
エグゼクティブサマリー（共通骨格・冒頭）
  ↓
本体セクション群（動的に設計）
  ↓
情報の矛盾・不確実な点（共通骨格）
追加調査が必要な領域（共通骨格）
結論（共通骨格）
出典（共通骨格）
```

## 原則8: 見出しは内容を特定する

見出し（章・小節）は、その節が答える問い、または扱う対象を特定できる具体的な句にする。原則1（各章は独立した問いに答える）が「章の役割」を定めるのに対し、本原則は「見出しの書き方」を定める。

- **情報量のない見出しにしない**: 「関連情報」「その他の論点」「補足」「〜を整理する」のような、中身を特定しない見出しを避ける。その節が答える問いか、扱う対象を見出しにする
- **見出しを節の結論の「セリフ」にしない**: 見出しの時点で読者がオチを知る状態を避ける。見出しは対象や問いを指し、結論は本文で示す（例: ✕「半導体規制は中国に効かない」→ ○「半導体規制の対中効果」）
- **区切り線やダッシュで二要素を詰め込まない**: 「種別──主題」「主題──概念」のように記号で2要素を並べない。単一の自然な句にする（`tone-rules.md` の整形ルールと整合）
- **疑問形でも名詞句でもよい**: 見出しが問いを示すのは可（例: 「地政学的リスクはどの類型に分けられるか」）。これは本文での読者への問いかけ・反語の禁止（`tone-rules.md` 機能1）とは別レイヤーである。見出しが対象・問いを指していれば、疑問形か名詞句かはテーマのトーンで選ぶ

## 構造改訂時の追加原則

フェーズ3（ステップ11）で仮構造を改訂する際は、上記の原則に加えて以下を考慮する。

### 改訂の判断基準

| 状況 | 改訂アクション |
|---|---|
| 予想外の重要論点が見つかった | 章を追加する |
| 情報が不十分で薄い章がある | 関連する章に統合する |
| 1章の情報量が多すぎる | 章を分割する |
| 調査結果から、より自然な因果の流れが見えた | 章の順序を変更する |
| 当初想定と異なるデータや視点が得られた | 分析要素を変更する |
| 定量データが想定より多い/少ない | 定量/定性比率を調整する |
| 調査結果が当初の問いを修正する | 中心主題を微調整する |

### 改訂の制約

- 改訂は全調査完了後に**1回のみ**実施する
- 共通骨格の要素は改訂で削除できない
- 改訂理由を `structure.md` に必ず記録する
- 改訂後の構造も原則1〜8を満たしていることを確認する
