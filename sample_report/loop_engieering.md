# Loop Engineering: AIコーディングエージェント開発における「ループを設計する」パラダイム

**作成日**: 2026-07-03 JST

## エグゼクティブサマリー

本レポートの主題は、2026年半ばにAIコーディングエージェント開発の語彙として急浮上した「Loop Engineering（ループエンジニアリング）」とは何であり、なぜこの時期にパラダイムとして成立し、実務者は何をどう設計すべきか、である。

Loop Engineering は、エンジニアが「エージェントに1ターンずつプロンプトする人」であることをやめ、「エージェントにプロンプトするシステム（ループ）」を設計する実践を指す。Google Cloud AI のディレクターであった Addy Osmani が2026年6月7日のエッセイで命名し、Peter Steinberger（OpenClaw 作者）が「エージェントにプロンプトするのはもうやめ、プロンプトするループを設計せよ」の一文に圧縮し、Boris Cherny（Anthropic の Claude Code 責任者）が「私の仕事はループを書くこと」とベンダー側から追認したことで、約1ヶ月で拡散した [^1][^2]。

調査から3つの発見が得られた。第一に、これは無からの発明ではない。技術的実体は ReAct（2022）、AutoGPT（2023）、Ralph technique（2025年7月）と積み上げられてきた反復エージェントの系譜であり、2026年に、モデルの信頼性、コンテキスト窓、トークン単価という3条件が揃って初めて経済的に成立し、命名が追いついた [^3][^4]。第二に、ループの価値は反復そのものでなく、検証器（verifier）と停止条件（stop rules）にある。ほぼ全ての解説が見落とすが、ボトルネックはモデルでなく「何をもって『良い』『完了』とするか」を定義する検証にある [^5]。第三に、最大のリスクは止まらないループとトークン暴走であり、Stanford Digital Economy Lab の研究は、エージェント的コーディングがコード推論やチャットの1000倍のトークンを消費し、同一タスクでも最大30倍ばらつくことを示した [^6]。

結論として、Loop Engineering はプロンプトエンジニアリングを置換する新レイヤーであると同時に、その大半は既存のオーケストレーション実践の再命名でもある。確立された部分（検証設計・停止条件・トークンガード）は本番運用に必須の規律であり、誇張された部分（全自動化の万能感）はコストと理解負債の現実に直面する。実務者の役割は Operator から System designer へ移り、希少スキルは「良い」と「完了」の定義に移動する。

## Loop Engineering の定義と起源

Loop Engineering の中核定義は提唱者本人の言葉に集約される。「Loop engineering は、エージェントにプロンプトする人としての自分自身を置き換えることだ。代わりにそれを行うシステムを設計する。ここでのループは、目的を定義しAIが完了まで反復する再帰的なゴールと考えられる」[^1]。旧来のワークフロー（プロンプトを書く→出力を読む→次のプロンプトを書く）を、仕事を発見し、エージェント（しばしばサブエージェント）にタスクを渡し、結果を検証し、状態を永続化し、次の行動を決める小さな自律制御システムの構築へ置き換える。それをスケジュールで、あるいはゴール達成まで回す [^2]。

命名の経緯は明確に特定できる。Addy Osmani が2026年6月7日にエッセイ「Loop Engineering」を公開し（O'Reilly Radar にも転載）[^1][^7]、同じ週に Peter Steinberger が「コーディングエージェントにプロンプトするのはもうやめるべきだ。エージェントにプロンプトするループを設計すべきだ」と要約した [^2][^5]。Anthropic で Claude Code を作った Boris Cherny は「もう Claude にプロンプトしない。Claude にプロンプトし次に何をするか判断するループを走らせている。私の仕事はループを書くことだ」と述べ、ツールを作る側からも同じ結論が示された [^2][^5]。swyx はこれを「loopcraft: ループを積み重ねる技法」というほぼ同義の別称で框組みした [^8]。約1ヶ月のうちに LangChain、Kilo、MindStudio、O'Reilly などが相次いで解説を出し、用語として定着した [^8][^9]。

Loop Engineering の位置づけは、隣接概念との3層関係で理解できる。Addy Osmani 自身が整理する通り、最下層に agent harness engineering（単一エージェントが1回動く環境そのものを整える設計）があり、その全体像として factory model（ソフトウェアを作るシステム）がある。Loop engineering は harness の1階上に座る。「ハーネスが単一エージェントの実行を装備するのに対し、ループはタイマーで動き、小さなヘルパー（サブエージェント）を生成し、自らにフィードバックし続けるもの」である [^1][^2]。Cobus Greyling はこれを進化の系列として描く。「Context Engineering が新しかった時代、次に Harness Engineering、そして今 Loop Engineering」[^2]。つまり Loop Engineering は既存の下位レイヤーを置換するのでなく、その上に積まれた制御レイヤーである。

この層の区別は本レポートを通じて重要になる。後述するトークン暴走や理解負債といった失敗モードの多くは、ループ層の設計（停止条件・検証・予算ガード）の欠如から生じるのであって、ハーネスやモデルの欠陥ではないからである。

## ループの解剖学

Loop Engineering を成立させる「ループ」は、単一の概念でなく3つの視点で分解できる。単一ターン内で回る基本サイクル、ループを設計するときに決める4要素、そして複数のループを積み重ねるスタックである。

### 単一ターンの基本サイクル

最も基礎的なループは、単一のエージェントターン内で harness が回す反復サイクルである。Oracle のエンジニアリング解説はこれを次のように定義する。「エージェントループは、harness が単一エージェントターン内で回す反復サイクルだ。①実行コンテキストを組み立て（システム指示・会話状態・取得したメモリ・ツール出力・関連する外部データ）、②推論モデルを呼び出して次に何をするか決め、③行動し（ユーザーへの応答・ツール呼び出し・メモリや状態の書き込み・計画の更新）、停止条件が実行を終えるまで繰り返す」[^10]。各サイクルは自らのトレース（アシスタントのメッセージ、ツール出力、状態更新）をコンテキストに追記し、次の反復の入力にする。一度処理して静的な応答を返すのでなく、出力を自らに供給して推論・行動・観測・再推論を完了まで続ける点が、通常の一問一答との違いである [^10]。

Oracle はこのループを3レベルで整理する。レベル1は多くの開発者が最初に出会う最小ループで、LLM と少数のツールと応答からなる。レベル2はループ内にライフサイクルを導入し、メモリ操作によってステートレスなプロセスを状態を持つ推論エンジンに変える。レベル3はループの内側と外側の両方に操作を広げ、harness そのものが独立したシステムになる [^10]。このレベル3が、Loop Engineering が対象とする領域である。コンテキストウィンドウの圧迫と運用上の安全（タイムアウト、反復上限、予算ガード）が後付けでなく第一級の関心事として設計に組み込まれる点も、この定義の特徴である [^10]。

### ループ設計の4要素

ループを設計するとき、実際に決めるのは4つの要素である。本番運用を扱うガイドはこれを trigger（起動）、topology（構成）、verifier（検証器）、stop rules（停止条件）と整理する [^5]。この4要素が、プロンプトエンジニアリングとの違いを最も鋭く示す。「良いプロンプトは1ターンを直すだけだ。次にどのターンが来るか、何をもって完了とするか、不可逆な行動の前に誰が承認するか、支出をどう上限内に保つかについては何も言わない。それらはループの問いであり、その答えがエージェントシステムが実ワークロードとの接触に耐えるかを決める」[^5]。

trigger は何がループを起動するか（手動、スケジュール、イベント）を、topology は単一エージェントかサブエージェント群かオーケストレーションかを、verifier は出力が基準を満たすかを誰がどう判定するかを、stop rules はいつ止めるかを定める。この4要素のうち、後述する通り verifier と stop rules が実務上のボトルネックになる。

### ループのスタック

単一のループは出発点にすぎない。swyx が「loopcraft: ループを積み重ねる技法」と呼んだ通り、複数のループを積み重ねて効果的なエージェントを組む [^8]。LangChain はこのスタックを4層で示す [^8]。

| レイヤー | 何をするか | 効果 |
|---|---|---|
| 1. エージェントループ | モデルがタスク完了までツールを反復呼び出し | 作業の自動化 |
| 2. 検証ループ | 出力をルーブリックで採点し、失敗ならフィードバック付きで再試行 | 品質と正しさの担保 |
| 3. イベント駆動ループ | イベントがエージェント実行を起動し実システムを更新 | スケールでの自動作業 |
| 4. ヒルクライミングループ | 本番トレースを分析エージェントに供給し harness 構成を改善 | ハーネス自体の改善 |

この表から読み取れるのは価値の重心の位置である。LangChain は「ループ1と2は以前から考えてきたが、焦点はループ3と4へ移すべきだ。そこでは、あなたの基準に応じて継続的に改善するエージェントをエコシステムに埋め込むことで、価値が複利的に積み上がる」と述べる [^8]。単発の自動化（ループ1）から、システムが自らを改善する学習ループ（ループ4）へ進むほど、Loop Engineering の戦略的価値は増す。この「複利」の含意は組織レベルの示唆（後述）に直結する。

### 観測可能性と状態の永続化

ループが機能するかは観測の質で決まる。「ループはその観測と同程度にしか良くならない。エージェントが関連するテストを走らせ、コンパイラ出力を見て、UI を検査し、ログを読めなければ、部分的に盲目で動作している」[^9]。良い観測可能性は、高速で標的を絞ったテスト、型チェックと lint、ビルドコマンド、ランタイムログ、スクリーンショットやブラウザ確認、そしてコードレビューコメント（人間の判断）から構成される [^9]。これらは verifier に「証拠」を供給する装置であり、曖昧な自信を検証可能な根拠に変える。

状態の永続化も基本サイクルの延命に効く。長い単一セッションはコンテキスト窓が古い推論や行き止まりで埋まって劣化するため、進捗を会話履歴でなくディスクやファイルシステムに逃がす設計が採られる。この「コンテキストのリセット」という発想は、次章以降で見る Ralph loop の中核であり、なぜ2026年にループが実用化したかという因果とも結びつく。

## なぜ2026年にシフトが起きたか

Loop Engineering は新しい発明ではなく、既存の反復エージェントに名前が付いた現象である。だからこそ「なぜ2026年半ばに」という問いが重要になる。答えは、供給側の技術条件が揃ったことと、需要側で全チームが同じ壁にぶつかったことの合流にある。

技術的な系譜は4年に及ぶ。エージェントを推論と行動のループとして回す発想は ReAct（2022）に遡り、AutoGPT（2023）が自律実行を大衆化し、Geoffrey Huntley の Ralph technique（2025年7月）が実務的なパターンを確立し、2026年にコーディングエージェント製品へネイティブ機能として搭載された [^3][^4]。この系譜の上で命名レイヤーも進化した。プロンプトエンジニアリング（2023〜24）、コンテキストエンジニアリング（Anthropic のエッセイが画期）、ハーネスエンジニアリング、そして Loop Engineering という順で、抽象化の階層が一段ずつ上がってきた [^2][^11]。

シフトを成立させた直接の因果は、3つの供給側条件の同時成立である。ある解説はこれを端的にまとめる。「モデルがついに十分信頼できるようになり、コンテキストウィンドウがついに十分大きくなり、コストがついに十分下がった。これがループを回すことを、科学実験でなく経済的に正気の沙汰にした」[^4]。個々の条件は単独では不十分だった。信頼性が低ければ無人実行はすぐ破綻し、コンテキスト窓が小さければ長いタスクを保持できず、単価が高ければ反復が経済的に破産する。3条件が同時に閾値を超えたことで、無人で回すループが初めて実用の領域に入った。

需要側の要因も揃っていた。「2026年半ばまでに、AIコーディングエージェントは数時間にわたり多段タスクを自律実行できるほど有能になった。ボトルネックはモデルから移った」[^11]。プロンプトの言い回しが最適化しつくされ、1ターンの品質改善では埋められない問題（次に何をするか、いつ完了か、誰が承認するか、支出上限)が前景化した。Loop Engineering という語が急速に広まったのは、まさに「本番でエージェントを走らせる全チームが既にぶつかっていた壁に名前を付けた」からだと分析される [^5]。

ただし、これはプロンプトエンジニアリングの消滅を意味しない。両者は置換でなく積層の関係にある。「ループを設計する前に、モデルに実際何ができるかを理解する必要がある。プロンプトエンジニアリングはモデルの能力を探る方法だ」[^12]。ループはプロンプトの上位に乗る制御レイヤーであり、下位のプロンプト品質が悪ければループ全体が破綻する。この積層関係を見誤ると、後述する「偽物のループ」（劣悪なプロンプトを無限に再試行してトークンを浪費するだけの構造）に陥る。

## 実装パターンとツールエコシステム

Loop Engineering は哲学として語られがちだが、実装の実体は具体的である。中核となる現場発のパターン、製品に搭載されたネイティブ機能、サブエージェント構成、停止条件の実装方式、そしてタスク選別の基準の順に見る。

### Ralph loop

Loop Engineering の直接の祖先であり最も明快なパターンが、Geoffrey Huntley が2025年7月に公開した Ralph technique（The Simpsons の Ralph Wiggum に由来）である。仕組みは拍子抜けするほど単純で、コーディングエージェントを素の bash while ループの中で回す。書かれた仕様に対して同じプロンプトを与え、エージェントに1つのタスクを選ばせて実装させ、それが終わったら完全に新しいインスタンスを起動して同一のプロンプトを再投入する。これを作業が完了するまで繰り返す [^3][^13]。

Ralph の非自明な洞察は、ループそのものでなくコンテキストのリセットにある。長いエージェントセッションはコンテキスト窓が古い推論・行き止まり・陳腐化したファイル内容で埋まるにつれて劣化する。Ralph は毎回クリーンに始め、進捗を成長し続ける会話でなくディスクと git に残す。つまり会話履歴でなくファイルシステムをメモリとして使う [^3][^13]。Huntley 自身の表現では「予測不能な世界における、決定論的なほど単純なもの。動かなそうに見えて、動く」[^3]。この設計は前章で見た「なぜ今か」と直結する。コンテキスト劣化という制約を、窓を大きくするのでなくリセットで回避する発想であり、モデルが十分安価だからこそ「毎回作り直す」冗長性が経済的に許容される。

Ralph は運用姿勢も規定する。Huntley は「あなたの仕事は今やループの上に座ることだ。中でなく」と述べ、失敗ドメインを見つけたらエンジニアの帽子をかぶって二度と起きないよう解決し、プロンプトを「ギターのようにチューニング」して収束させると説明する [^3][^14]。エージェントを作ること自体は難しくない。「300行のコードをループで回し LLM トークンを注ぎ込めば、エージェントの出来上がりだ」[^3]。難所は作ることでなく、ループを監督し、失敗パターンを潰し、止め時を決める側にある。

### ネイティブプリミティブの収束

2026年5月までに、このパターンは手書きの bash スクリプトから製品そのものへ移った。Claude Code はネイティブの `/goal` コマンドをバージョン 2.1.139 で出荷し、OpenAI の Codex CLI も数週間後に自版を出荷したとされる [^4]。ただしこの具体的なバージョン番号は単一の二次ソースにのみ依拠しており、一次確認は取れていない（報道ベースの記述として扱う）。

注目すべきは、競合する2つの製品が極めて似たプリミティブに収束した点である。Claude Code は `/loop`、`/goal`、スケジュールタスク、hooks、worktrees、skills、サブエージェント、MCP を備え、Codex は Automations、`/goal`、agent skills、サブエージェント（`.codex/agents/`）、connectors を備える。「形は両者で同じだ。ループを一度設計すれば、どちらのハーネスでも走る」[^15][^1]。この収束は「ループの形」がツール非依存の抽象になりつつあることを示す。実務的には、特定ベンダーへのロックインを過度に恐れずループ設計の技能に投資できるという含意を持つ。

### サブエージェントとオーケストレーション

topology の選択肢の中心がサブエージェントである。Codex ではエージェントを `.codex/agents/` の TOML ファイルとして定義し、それぞれに名前・説明・指示、そして任意でモデルと推論努力（reasoning effort）を指定する。これにより、セキュリティレビュー担当は高努力の強力なモデルに、探索担当は高速で読み取り専用の軽量なものに、と役割ごとに使い分けられる。Claude Code も `.claude/agents/` で同様のことを行い、作業を受け渡すエージェントチームを組む。両者に共通する典型的な分業は、1体が探索し、1体が実装し、1体が仕様に対して検証する構成である [^1]。この「探索・実装・検証」の三分業は、前章のループ設計4要素のうち topology と verifier を具体化したものにあたる。

### 停止条件の実装バリエーション

stop rules は実装方式が複数あり、タスクの性質で選ぶ [^16]。

| 方式 | 「完了」をどう判定するか | 適するタスク |
|---|---|---|
| Ralph ループ | 素の while ループが同じプロンプトを再発火。賢い「完了」判定はなく、あなたが停止ボタン | 機械的で反復的なタスク。粗いが頑健 |
| Claude Code `/goal` | 小型で高速なモデルが毎ターン後に停止条件を判定 | 曖昧な「完了」で、なお評価が要るもの |
| ゴール追跡設定 | エージェントが自分の進捗をファイルで追跡し、「完了」を事前定義 | 方向維持が必要な長時間走行 |
| 自己ホスト（Hermes 型） | 自前インフラで継続実行し、セッションを跨いで状態を保持 | 一度設定し放置する常時稼働エージェント |

この表から、停止判定の「知能」の度合いがコストと信頼性のトレードオフになることが読み取れる。Ralph のように人間が停止ボタンになる方式は頑健だが人手を要し、`/goal` のように小型モデルに判定させる方式は自動だが判定誤りのリスクを負う。Kilo は有用な停止規則として、要求された挙動が実装され検証が通ったら止める、認証情報やデータや製品判断を要するブロッカーで止める、明示承認なしに破壊的コマンドの前で止める、無関係なユーザー変更に影響するなら止める、タスク範囲外なら止める、を挙げる [^9]。停止条件は「いつ完了か」だけでなく「いつ人間に委ねるか」の設計でもある。

### タスクがループ型かの判定

全てのタスクがループに値するわけではない。firecrawl は候補を3つのチェックにかけることを勧める。Repetitive（システムを設計するコストを回収できるほど頻繁にやるか）、Reviewable（「完了」をエージェントか検証器サブエージェントが実際に走らせられるチェックとして表現できるか。何が合格かを定義できなければループは止め時を知らない）、Valuable（出力がトークン代に見合うか。ループには時間と金の下限コストがあり、些末な作業はそれを超えない）である [^15]。3つ全てを満たすならループが適し、1つでも欠けるなら手動プロンプトか通常のスクリプトの方が良い。この選別基準は、次章の失敗モード（特にトークン浪費）を未然に防ぐ第一のフィルタである。

## 限界・失敗モード・批判

Loop Engineering への評価は割れている。提唱者の Addy Osmani 自身が「これがコーディングエージェントとの働き方の未来かもしれないと信じる一方、まだ早期で懐疑的だし、トークンコストには絶対に注意しなければならない」と留保を付けている [^1]。批判は主に4つの軸に整理できる。

### トークン経済と暴走リスク

最も具体的で定量的な批判はコストである。Stanford Digital Economy Lab の研究「How Do AI Agents Spend Your Money?」（著者に同ラボのディレクター Erik Brynjolfsson、ファカルティリード Sandy Pentland、ポスドクの Jiaxin Pei を含む）は、エージェント的タスクが「独特に高価で、コード推論やコードチャットの1000倍のトークンを消費する」ことを見出した [^6]。さらに、エージェントが自分の支出を予測できない理由として、どれだけコンテキストを蓄積するか事前に分からないこと、エージェントの軌跡が「本質的に確率的」であることを挙げる。「同じエージェントを同じタスクで走らせても、コストは最大30倍ばらついた」[^6]。モデルは一貫してトークン消費を過小評価し、人間の直感も当てにならない。タスクの難易度がエージェントのコストと一致しないためである [^6]。

この定量が失敗モードの物理的裏付けになる。ハードなトークン/ドル予算、無進捗チェック、明示的な反復上限がなければ、エージェントは週末で請求サイクルを焼き尽くしうる。実際、コミュニティでは「一晩で数百ドルを静かに食ったループの領収書」が共有され、どのスレッドでも開発者が最初に挙げる懸念はアーキテクチャでなくトークンである [^15]。ある実務者はこれを鋭くまとめる。「ハードなトークン予算と厳格な終了条件のない自律ワークフローは、エンジニアリングではない。監視されない無限ループだ」[^17]。処方は地味だが明快で、「全てのループはハードガードと共に出荷する。さもなくば出荷しない」[^15]。なお、Stanford 論文の数値は媒体により振れがある。OpenReview に掲載された草稿版は「最大10倍」のばらつきと記載しており、公開された writeup と arXiv 版の「30倍」との間に版差異がある [^6][^18]。

### 停止問題と理解負債

コストの根底には、より本質的な問題がある。「Loop Engineering の課題は推論ではない。停止することだ」[^19]。数十年にわたりソフトウェア技術者は無限ループを避けてきたが、Loop Engineering はある意味で意図的に無限ループを作り、そこに賢い停止判定を後付けする。firecrawl は最大リスクを「止まらないループ」と名指しし、第2のリスクとして「理解負債（comprehension debt）」を挙げる。「あなたが書いておらず理解もしていないコードがリポジトリに入る。ループはその検証ステップと停止条件と同程度にしか良くならない」[^15]。

理解負債は、生産性の見かけと保守性の実態の乖離として現れる。エージェントが大量のコードを生成し検証も通るが、人間の誰もその設計意図を把握していない状態は、短期の速度と引き換えに長期の保守コストを積み上げる。これはトークンコストのように請求書に現れないため、より静かに蓄積する。オーケストレーション税とも呼ばれるレビュー帯域の限界がここに重なる。エージェントを増やすほど、人間が結果を確認できる能力が律速になり、「理解できない静かな成功」がリスクとして残る [^20]。

### buzzword論争と既視感

用語そのものへの懐疑も根強い。「Loop Engineering は次のAI開発 buzzword か」という Reddit スレッドで、開発者は「これは本物の次の抽象化レイヤーだ」と「帽子をかぶった cron ジョブにすぎない」の間で二分する [^15][^21]。既視感の指摘は具体的である。「ループも我々にとって新しくない。CI/CD、GitFlow など、結果を生むために作ってきた仕組みの大半はループの上に築かれている。人間を中に入れて」[^17]。

この批判に対する反論は、意思決定者の有無に置かれる。ループが単なる cron ジョブと異なるのは、状態を読み、行動し、検証し、反復する意思決定者を含む点だとされる [^20]。ただし既視感批判の核心はより深い含意を持つ。「我々はより良いループを必要としているのでなく、より少ないトークンで良い結果に至る、より良いモデルとエージェントを必要としている」[^17]。つまり Loop Engineering は本質的な効率問題を解くのでなく、モデルの非効率を反復と検証で覆い隠す対症療法にすぎないという見方である。この見方に立てば、モデルが十分賢く安価になれば、精緻なループ設計の必要性そのものが縮小しうる。

### 本物のループと偽物のループ

これらの批判を踏まえると、価値あるループと無価値なループの線引きが実務上の焦点になる。ある実務者は Loop Engineering が「本物か」を実測で検証した。「オンラインで『エージェントループ』と呼ばれるものの大半は、チャットボットが while ループの中で自分自身を呼んでいるだけだ。何かを尋ね、答えを得て、その答えを戻し、繰り返す。これはエンジニアリングではない。モデルが自分のミスを、時間でなくトークンで払う、高価な試行錯誤だ」[^22]。

対照的に価値あるループは、「モデルの推論が一度だけ起き、固定され検査可能な何かにロックされ、その後は一切モデルを呼ばずに走る」構造だという。実測では、あるタスクを初めて構築するのにフロンティアモデルを通しで使って6分49秒かかり、実際のクレジット残高を相応に消費した。しかし保存後に同じタスクを再実行すると、5秒で完了し、残高は前後で同一、つまり消費ゼロだった [^22]。この差が「トークン効率」をスローガンから検証可能な数字に変える。ただし限界もある。「今日のトップAIニュースを要約する」ように毎回新鮮な判断を要するタスクは、内容が日々変わる設計である以上コンパイルできず、無料化できない。そうしたタスクの設定画面にトークンコストの予告が一切ないことが、「このカテゴリは固定スクリプトのように無料にできない」という静かな告白だと指摘される [^22]。この線引きは、前章の「タスクがループ型かの3チェック」の Reviewable 基準と一致する。推論を一度固定でき検証可能なタスクこそがループに値する。

## 実務者への含意

Loop Engineering が本物である部分と誇張である部分を切り分けたうえで、実務者への含意を対象者別に見る。共通する変化は、価値の重心が「良いプロンプトを書く技能」から「良いと完了を定義し検証する技能」へ移ることである。

役割の進化は次のように整理される [^16]。

| 年 | 時代 | やること | 役割 |
|---|---|---|---|
| 2024 | プロンプトエンジニアリング | 良いプロンプトを書き、良い出力を得る | Operator |
| 2025 | 並列エージェント | 1つのチャットのお守りをやめ、複数を同時に走らせる | Manager |
| 2026 | Loop Engineering | エージェントを走らせるループを作る | System designer |

この表の含意は、Claude Code を作る人々の証言が「静かな部分を声に出した」ときに露わになる。ある朝には彼らはもうプロンプトを書いておらず、別のモデルが書いており、彼らは数百から数千のエージェントを同時に管理している。「もしあなたが毎ステップでエンターを押していないなら、何かが『十分に良いか』を判断しなければならない。その何かがゲームの全てだ」[^16]。

### 個人開発者

個人開発者にとっての第一の転換は姿勢である。Huntley の「ループの中でなく上に座る」が要点で、実行そのものはエージェントに委ね、失敗ドメインの発見と恒久対策、プロンプトの収束チューニング、止め時の判断に人間の時間を移す [^3]。第二に、新たな律速はオーケストレーション税、すなわちレビュー帯域である。エージェントを増やすほど確認能力が追いつかなくなるため、個人が扱えるループ数には検証能力の上限がある [^20]。したがって個人開発者への処方は逆説的で、ループを増やす前に検証を自動化し（テスト・lint・スクショ）、自分がレビューしきれる範囲にループ数を絞ることである。Louis-François Bouchard は5つの構成要素（automations、worktrees、skills、connectors、sub-agents）と2つの罠（曖昧なゴール、暴走するトークンコスト）を挙げ、「いつプロンプトし、いつループするかを知り、どちらの場合もエンジニアであり続ける」ことを説く [^20]。

### チームとコードベース

チームにとって Loop Engineering は、コードベースそのものを「ループ対応」にする整備作業を要求する。AI Builder Club はこれを2つの性質に整理する [^16]。Legible（エージェントが何をどこで変えるべきか見つけられる）ためには、AGENTS.md や CLAUDE.md を約100行のインデックスに保って深いドキュメントへ誘導し（OpenAI は自社のものを約100行に保つ）、守らせたいルールをカスタム lint に焼き込む。たとえば触ってほしくないレガシーフォルダからの import を lint で失敗させれば、エージェントが正しいドキュメントを読むことを祈る代わりに自動で警告が出る [^16]。Executable（エージェントがほぼゼロのトークンコストで作業を開始できる）ためには、dev サーバを起動済みにする `dev local` スクリプトを書き（毎回3〜5分のアプリ起動でトークンを焼かないように）、リポジトリを worktree フレンドリーにして5つの並列エージェントが衝突せず各自のサーバを立てられるようにする [^16]。

より本質的なのは検証システムの設計である。「Loop Engineering における最大の間違いは、悪いコードを書くエージェントを作ることではない」。信頼できる検証システムを設計できないことである [^23]。verifier がボトルネックであってモデルではないという認識が、チームの投資配分を決める。テストカバレッジ、評価データセット（evals）、レビュー自動化への投資が、より良いプロンプトへの投資より優先度が高い。

### 組織

組織レベルの含意は、Loop Engineering の価値がループスタックの上位層で複利的に増すという構造（前述のループ4）から導かれる。LangChain が引く Satya の框組みでは、「学習ループを早く作り、人間の判断とトークン資本が複利で積み上がる企業は、複製困難な優位を築く」[^8]。ここで組織の意思決定は2つになる。第一に、いつどの業務にループを導入するかの選別（トークン資本の配分）。Stanford の研究が示す通りコストは予測困難で最大30倍ばらつくため、予算ガードとコスト観測を組織の標準運用に組み込む必要がある [^6]。第二に、学習ループ（本番トレースから harness を自動改善するループ4）の早期構築である。これは単発の効率化でなく、組織固有の判断基準がエージェントに蓄積される資産形成であり、後発が模倣しにくい。

Andrew Ng は0-to-1プロダクト構築に3つのループを使うと述べる。エージェント的コーディングループ（プロダクト仕様と任意で評価データをエージェントに与える）、開発者フィードバックループ、外部フィードバックループである。「ループを閉じるこの考えは昨年末に離陸し、人間の介入なしにコーディングエージェントを長く生産的に働かせる点でゲームチェンジャーだった」[^24]。組織にとって Loop Engineering は個人の生産性ツールでなく、人間の判断を検証基準としてシステムに埋め込み続ける組織能力の問題である。

## 情報の矛盾・不確実な点

- **Claude Code のバージョン番号（v2.1.139）と Codex の出荷時期**: `/goal` コマンドが Claude Code v2.1.139 で出荷され Codex CLI が数週間後に続いたという記述は、単一の二次ソース（rzp / young urban project）にのみ依拠する。事実抽出系（具体的バージョン番号）でありながら一次（公式リリースノート）未確認のため、矛盾の有無に関係なく低確信として扱う。本文では報道ベースの記述に限定した [^4]。
- **Stanford 論文のばらつき数値（30倍 vs 10倍）**: Stanford Digital Economy Lab の公式 writeup と arXiv 版は同一タスクで「最大30倍」とするが、OpenReview 掲載の草稿版は「最大10倍」と記載する。版差異があり、確定値として単一化できない [^6][^18]。1000倍（対 code reasoning/chat）は複数経路で一致し確度が高い。
- **提唱者の原発言（ツイート）の直接確認**: Boris Cherny と Peter Steinberger の発言は Addy Osmani のエッセイと Cobus Greyling の解説を経由した引用で、原ツイート（x.com）そのものには本調査では直接到達していない。発言内容は複数の二次ソースで一致する。
- **「Ralph 2025年7月」の日付**: Ralph technique の考案者 Huntley 本人のサイトは取得したが、「2025年7月公開」の日付は二次ソース（rzp, LinearB 等）に依拠する。

## 追加調査が必要な領域

- **定量的な採用実態と ROI**: Loop Engineering を導入したチームの生産性向上や、理解負債・保守コストの増分を計測した実証データは本調査では見つからなかった。逸話的なコスト報告（一晩数百ドル）はあるが、体系的な費用対効果の研究が要る。
- **セキュリティと不可逆行動のガバナンス**: 自律ループが不可逆な行動（デプロイ、データ削除、外部API課金）を取る際の承認設計は、Kilo の停止規則等で断片的に触れられるのみで、体系的なガバナンスフレームワークの調査が要る。Anthropic の2026 Agentic Coding Trends Report は dual-use リスクとセキュリティファースト設計に言及しており、追加調査の価値がある [^25]。
- **モデル進化による陳腐化の可能性**: 「より良いループでなくより良いモデルが必要」という批判が正しければ、モデル効率の向上で Loop Engineering の必要性がどう変化するか。この反実仮想は本調査の範囲では評価しきれない。
- **日本語圏・非英語圏での受容**: 本調査で到達したソースは英語圏に集中した。日本の実務コミュニティでの受容度や独自の実践は追加調査が要る。

## 結論

本レポートの主題は、Loop Engineering とは何であり、なぜ2026年に成立し、実務者は何を設計すべきか、であった。その答えは次の通りである。Loop Engineering は、エージェントに1ターンずつプロンプトする作業を、仕事を発見し検証し停止を判断する制御システムの設計へ置き換える実践であり、プロンプトエンジニアリングを消すのでなくその上に積まれた新しいレイヤーである。2026年6月に Addy Osmani が命名したが、技術的実体は ReAct から Ralph loop に至る反復エージェントの系譜であり、モデル信頼性・コンテキスト窓・トークン単価の3条件が同時に閾値を超えたことで初めて経済的に成立した。

調査で最も重要な発見は3点である。第一に、ループの価値は反復でなく検証器と停止条件にあり、ボトルネックはモデルでなく「良い」と「完了」の定義にある。第二に、Ralph loop が示す通り、核心はコンテキストのリセット（会話でなくディスクをメモリにする）という設計であり、これが Claude Code と Codex のネイティブ機能へツール非依存の形で収束した。第三に、最大の失敗モードは止まらないループとトークン暴走であり、Stanford の研究はコストが通常の1000倍に達し同一タスクで最大30倍ばらつくことを定量的に裏付けた。

この変化の意味は関係者によって異なる。個人開発者にとっては、実行から監督へ役割が移り、レビュー帯域が新たな律速になる。ゆえにループを増やす前に検証を自動化することが合理的である。チームにとっては、コードベースを Legible かつ Executable に整備し、より良いプロンプトより信頼できる検証システムへ投資することが優先される。組織にとっては、トークン資本の配分と予算ガードが経営課題になり、本番トレースから自らを改善する学習ループの早期構築が模倣困難な優位を生む。

注視すべき先行指標は次の3つである。ネイティブループ機能（`/loop`、`/goal`、Automations）が両ベンダーで標準搭載され続けるかというツール収束の持続、Stanford 論文のようなトークン経済の実証研究が示すコスト曲線の変化、そして「より良いループより良いモデル」という批判が現実になるか、すなわちモデル効率の向上が精緻なループ設計の必要性を縮小させるかである。

短期から中期の展望として、現時点で利用可能な情報に基づくと、Loop Engineering は buzzword としての過熱を経て、その確立した中核（検証設計・停止条件・トークンガード・コンテキストリセット）が本番エージェント運用の標準規律として定着する可能性が最も高い。用語そのものが1年後も使われているかは不確実だが、それが名指した実践は既に全チームがぶつかった壁への必然的な応答であり、名前が廃れても実践は残る。前提条件として、現在のモデル単価とコンテキスト劣化の傾向が大きく変わらないことが必要である。モデルが大幅に安価になり長コンテキストで安定すれば、ループ設計の一部（特にコンテキストリセットや冗長な反復）は不要になり、Loop Engineering の重心は検証とガバナンスへさらに移動すると見られる。
